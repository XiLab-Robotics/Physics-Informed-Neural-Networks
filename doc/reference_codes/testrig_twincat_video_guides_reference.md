# TestRig TwinCAT Video Guides Reference

## Role In This Project

This note captures the repository-owned workflow used to analyze the local
video-guide bundle stored under `.temp/video_guides/`.

The goal is to convert that media bundle into reusable technical knowledge that
supports future TwinCAT deployment work for the TE models trained in this
repository.

The note complements, rather than replaces:

- `doc/reference_codes/testrig_twincat_ml_reference.md`
- `doc/technical/2026-03-26/2026-03-26-15-43-43_twincat_ml_export_and_testrig_reference_analysis.md`
- `doc/technical/2026-03-26/2026-03-26-15-59-15_post_campaign_twincat_deployment_evaluation_and_isolated_parallel_track.md`

## Why The Video Bundle Matters

The existing TwinCAT reference note already reconstructs the PLC code path from
the imported TestRig source.

The video-guide bundle adds a different class of evidence:

- operator-level explanations of how the TwinCAT workflow was assembled;
- screen-level demonstrations of the project structure and simulation flow;
- naming, path, and setup details that may not be obvious from code alone;
- errata that correct or refine the interpretation of the original walkthroughs.

This additional evidence is useful when deciding:

- how to prepare trained Python models for the current TwinCAT runtime contract;
- which workflow assumptions are merely paper-specific;
- how the TwinCAT code would need to change for different model families.

## Confirmed Companion-Note Facts Already Available

Even before full transcript extraction, the text files shipped beside the
videos already confirm several practical details.

### Simulation Columns

The companion note `Machine Learning - read me.txt` states that the simulation
workflow uses columns corresponding to:

- speed;
- torque;
- temperature;
- TE;
- absolute slow-shaft position in `[0, 360]`;
- cumulative slow-shaft position;
- cumulative fast-shaft position.

This is important because it extends the paper-level feature story with the
actual simulation-table schema used around the TwinCAT work.

### TE_Calc Errata

The companion errata note states that `TE_Calc` in TwinCAT is not simply a
Matlab-computed variable imported as-is.

Instead, the TE quantity is computed directly in TwinCAT from the recorded
total positions of the slow-shaft and fast-shaft motor side during an
experiment.

This correction matters for future integration because it makes clear that the
TwinCAT side already owns part of the TE reconstruction logic and should not be
treated as a passive display endpoint.

### Matlab Helper Artifact

The same errata note identifies `Extra_Data_For_Simul_TC_Cam` as the Matlab
file used to extract and combine:

- the ideal values generated by the cam profile;
- the real values recorded through TwinCAT.

This helper artifact is relevant for any future attempt to reproduce or extend
the simulation path shown in the videos.

## Repository-Owned Extraction Workflow

The repository now includes:

- `scripts/tooling/analyze_video_guides.py`
- `scripts/tooling/extract_video_guide_knowledge.py`

This script is designed to process `.temp/video_guides/` through four layers:

1. media inventory and metadata collection;
2. timestamped transcription with local Faster-Whisper models;
3. keyframe extraction through OpenCV;
4. quality-gated OCR on sampled TwinCAT-oriented screen regions when Tesseract is available locally.

The generated artifacts are written under:

- `.temp/video_guides/_analysis/`

This location intentionally keeps raw analysis outputs outside the canonical
documentation tree while preserving them in a reproducible repository-local
path.

The repository-owned per-video reports generated from those artifacts are now
stored under:

- `doc/reference_codes/video_guides/`

The current canonical promoted artifact set comes from the tracked remote-strong
rerun documented in:

- `doc/reference_codes/video_guides/REMOTE_HIGH_QUALITY_RERUN_README.md`
- `doc/running/remote_high_quality_video_rerun_status.json`
- `doc/running/remote_high_quality_video_rerun_checklist.md`

## Tooling Behavior

The analysis script is intentionally tolerant of partial environments.

It can still produce useful outputs when some optional backends are missing:

- if Faster-Whisper is unavailable, transcription is skipped explicitly;
- if OpenCV is unavailable, frame extraction is skipped explicitly;
- if `pytesseract` is installed but the Tesseract executable is absent, OCR is
  skipped explicitly.

This avoids silent failure and makes the environment limitations inspectable in
the generated inventory note.

## Recommended Usage

Install the optional video-analysis dependencies tracked in `requirements.txt`,
then run:

```powershell
python -B scripts/tooling/analyze_video_guides.py
```

For a smaller first pass on only the machine-learning videos:

```powershell
python -B scripts/tooling/analyze_video_guides.py --video-filter "Machine" --limit-videos 2
```

For a fast inventory-only pass without transcription:

```powershell
python -B scripts/tooling/analyze_video_guides.py --disable-transcription
```

## Initial Video-Derived Findings

The first repository-owned extraction pass was run on:

- `Machine_Learning_1.mp4`
- `Machine_Learning_2.mp4`
- `ML_Simulation_and_Generator_Cam.mkv`

The resulting transcripts remain an approximation because the source language is
spoken Italian with technical TwinCAT terminology, but the stronger ASR and the
quality-gated report layer are now materially better than the earlier raw
extraction pass and are usable as supporting engineering evidence.

### Runtime And Task Structure

The extracted transcripts repeatedly support the same architectural pattern
already seen in the imported PLC code:

- a dedicated machine-learning task around `500 us`;
- a faster companion task around `150 us`;
- inter-task communication through linked variables;
- non-negligible delay when values are sent from one task and consumed in the
  other.

This strengthens the conclusion that any future TwinCAT deployment path must be
evaluated not only for raw inference speed, but also for communication and
cycle-boundary delay.

### Saturation And Component Count

The first machine-learning video indicates that the faster task saturates when
too many harmonic components are handled directly there, which motivated moving
the ML work into a dedicated slower task.

This aligns well with the code-level observation that the deployed runtime uses
a reduced harmonic set rather than a large arbitrary basis.

### Predictor Interface Shape

The first video again points to the Beckhoff `FB_MllPrediction` method family
already seen in code, including:

- `Configure`;
- preferred-type inspection;
- model-name inspection;
- maximum-concurrency inspection;
- `PredictRef`;
- reset behavior.

The transcript also reinforces that `PredictRef` is operationally important for
the current file-handling strategy, which is consistent with the multi-engine
`engine_<n>` interpretation already documented in the canonical TwinCAT note.

### Input Ordering And Simulation Semantics

The simulation video and companion notes jointly reinforce the feature order
used by the current workflow:

- speed;
- torque;
- temperature.

The simulation transcript also indicates:

- torque sign is treated carefully, with a negative torque convention when the
  corresponding motor-side speed is positive;
- velocity used for ML input must remain explicit and is not interchangeable
  with every other motion quantity shown in the simulation;
- simulation support files are intended to reproduce the behavior of the
  machine-learning experiment rather than to replace the real TwinCAT
  experiment semantics.

### Package And License Preconditions

The simulation video also indicates a practical TwinCAT deployment precondition:

- the Beckhoff machine-learning package must be installed on the PC;
- for PLC execution, the package must also be installed on the PLC;
- the project-level license is attached on the TwinCAT side and then deployed
  with the project download.

This is an operational setup detail rather than a model-design rule, but it is
important for future export validation because a failed deployment attempt can
otherwise be mistaken for a model-format issue.

## Cross-Video Findings After The Remote-Strong 11-Video Rerun

The full promoted artifact tree now covers `11` source videos and provides a
meaningfully stronger evidence base than the initial partial extraction pass.

### 1. The Existing Runtime Is Structured, Not Monolithic

Across the machine-learning, simulation, and TestRig overview videos, the
deployment story is consistently structured around:

- generated prediction function blocks;
- explicit task assignment;
- CSV or table-fed operating variables;
- experiment-state orchestration;
- PLC-side post-processing or reconstruction logic.

This strengthens the repository conclusion that future TwinCAT-facing model work
should prefer structured or hybrid exports over opaque end-to-end black-box
stories.

### 2. Feature Semantics Stay Operationally Grounded

The video bundle repeatedly reinforces the operational importance of:

- speed;
- torque;
- temperature.

The simulation-focused videos further add practical surrounding columns for:

- TE;
- absolute slow-shaft position;
- cumulative slow-shaft position;
- cumulative fast-shaft position.

Future export planning should therefore keep the model-facing feature vector
explicitly distinct from the broader simulation or replay table.

### 3. Task Timing Is A Real Design Constraint

The machine-learning videos repeatedly support the same scheduling pattern:

- a dedicated ML-related task around `500 us`;
- a faster companion task for lower-level behavior;
- measurable inter-task communication delay;
- motivation to keep heavy work out of the faster path.

This means future TE models should be evaluated not only for fit quality, but
also for:

- schedulability;
- communication overhead;
- stability under task-to-task transfer delay.

### 4. `TE_Calc` Must Not Be Oversimplified

The errata and simulation videos materially strengthen the warning that
`TE_Calc` is not a trivial “Matlab output pasted into TwinCAT”.

The video evidence supports a more careful interpretation:

- TwinCAT/TestRig already owns meaningful TE-domain calculation,
  reconstruction, or mapping logic;
- variable mapping mistakes can invalidate the simulation or replay workflow
  even when the exported model itself is correct.

### 5. Experiment Logic Surrounds Inference

The video reports repeatedly show that the TestRig experiment path includes more
than inference:

- homing and zeroing;
- reset/error clearing;
- CSV read triggers;
- first-row initialization;
- mode-dependent or state-dependent activation;
- compensation application only in selected runtime conditions.

This means exported models must be designed against the real experiment contract
rather than against an abstract predictor interface.
