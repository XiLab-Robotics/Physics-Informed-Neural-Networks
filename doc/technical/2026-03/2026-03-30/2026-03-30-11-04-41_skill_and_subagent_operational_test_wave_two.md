# Skill And Subagent Operational Test Wave Two

## Overview

This document defines a second operational-test wave for the repository-owned
Codex skills and subagents that have not yet been exercised on a real task.
The target set is:

- `campaign-architect`
- `powershell-campaign-tooling`
- `scientific-reference-synthesizer`
- `repo-ml-reviewer`
- `repo-test-strategist`
- `twincat-deployment-analyst`

The goal is to validate that each artifact is useful on a realistic
repository-local task, produces non-redundant output, and improves execution
discipline or decision quality.

## Technical Approach

The operational test will be split into three bounded tracks so that each skill
or subagent is exercised against an appropriate repository surface without
touching protected files from the finished residual-harmonic campaign.

1. Campaign-review track
   - Use `campaign-architect` to review one existing completed campaign package
     for naming, artifact-layout, and planning/launcher alignment.
   - Use `powershell-campaign-tooling` to review the matching PowerShell
     launcher and launcher note for drift or fragility.
   - Produce a short repository-owned audit note with findings, if any, and a
     minimal fix set when local corrective edits are justified.

2. Reference-synthesis track
   - Use `scientific-reference-synthesizer` on a focused TE-modeling topic from
     `reference/` and matching repository summaries.
   - Produce an implementation-facing technical note that separates:
     reference-backed facts, repository consequences, implemented status, and
     open questions.

3. Subagent-review track
   - Exercise each repository subagent once on a scoped question that matches
     its intended role.
   - Integrate the outputs into a single repository-owned operational-test
     report that records usefulness, overlaps, and follow-up recommendations.

The default intention is to prefer review and synthesis over invasive
implementation. Small local fixes may be made when a concrete issue is found
and the fix is straightforward, but the wave is primarily a validation task for
the Codex artifacts themselves.

Planned subagent usage for the approved implementation:

- `repo-ml-reviewer`
  - Intended task boundary: review the selected campaign package and any
    resulting local fix set for workflow, naming, and regression risk.
  - Runtime launch still requires explicit user approval.
- `repo-test-strategist`
  - Intended task boundary: define the minimum safe evidence set for the
    campaign-review and reference-synthesis changes, including documentation QA
    and any report validation.
  - Runtime launch still requires explicit user approval.
- `twincat-deployment-analyst`
  - Intended task boundary: assess the deployment-facing implications of the
    selected reference-synthesis topic and whether the repository’s current
    modeling direction remains TwinCAT / PLC-friendly.
  - Runtime launch still requires explicit user approval.

## Implementation Steps

1. Inspect the current finished campaign state and choose a completed campaign
   package plus its launcher artifacts as the campaign-review target.
2. Run the `campaign-architect` workflow and record campaign findings.
3. Run the `powershell-campaign-tooling` workflow on the matching launcher and
   launcher note, then apply small local fixes only if warranted.
4. Choose a focused TE-modeling reference topic and run the
   `scientific-reference-synthesizer` workflow to produce a technical note.
5. Launch the approved subagents on their declared bounded scopes and collect
   their outputs.
6. Integrate findings into a repository-owned operational-test report covering
   usefulness, redundancy, and recommended next adjustments for the skills and
   subagents.
7. Run Markdown QA on touched Markdown files and perform any required report or
   PDF validation for artifacts generated by the wave.
