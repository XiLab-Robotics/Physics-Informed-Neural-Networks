# Campaign Launcher Short Command

## Overview

The current campaign runner works correctly, but the full launch command is long and easy to mistype when the user needs to rerun a specific campaign repeatedly.

This feature introduces a short launcher entry point for the Wave 1 recovery campaign, or a similarly reusable wrapper pattern, while preserving the existing terminal behavior:

- live training output must still stream to the console;
- per-run logs must still be written to the campaign log directory;
- campaign manifests and execution reports must still be generated by the existing runner;
- no training semantics should change.

The goal is convenience, not a new training system.

## Technical Approach

The preferred implementation is a thin wrapper around `scripts/training/run_training_campaign.py`.

1. Keep the existing runner as the execution engine.
2. Add a short launcher script or command alias that expands to the approved Wave 1 recovery campaign config list.
3. Forward all arguments directly to the runner so the current logging, progress output, queue handling, and artifact generation stay intact.
4. Keep the launcher repository-local and explicit, so the exact campaign configuration remains inspectable.

The wrapper should not suppress stdout, stderr, or Lightning progress bars. It should only reduce typing overhead.

## Implementation Steps

1. Add a compact launcher script or command wrapper for the Wave 1 recovery campaign.
2. Make the launcher call the existing campaign runner with the exact approved config list.
3. Preserve all terminal logging, progress output, and per-run file logging.
4. Document the new short command in the usage guide and in the campaign runner documentation.
5. Validate the launcher by running it once and confirming that the resulting campaign output matches the direct runner behavior.
